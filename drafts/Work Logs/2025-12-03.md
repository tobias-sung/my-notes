At this point, the SD card reader is working without any issue but there are still issues with formatting and content.

The first issue is that the SD card initialization occurs inside a task. Unlike the other hardware periperhals, it doesn't start until after the Task Scheduler begins. I don't know why it's designed like this, but trying to copying the initialization code into a functiona and then running that function before the Task Scheduler just results in the program hanging. 

The bad new is that this means all the debug messages generated before the task scheduler starts don't get written to the SD card.
****
I think the initialization functions were designed to work after the Task Scheduler starts, so there's nothing I can do short of rewriting the whole initialization procedure which I am incapable of doing.

So I've settled on a dumb method for now. I have a boolean flag, `startup_done` that is FALSE until the SD card task has completed all its initialization work. Then it gets set to TRUE. When `debug_print()` is run, it checks if `startup_done` is TRUE. If not, it saves its output to a really long character array, `startup_log`. 

```c
if (startup_done){
    xTaskNotify(sd_writeTaskHandle, 0, eNoAction);
} else {
    strcat(startup_log, debug_buffer);
}
```

Then, when the SD task starts, it first writes everything in `startup_log` into the text file before starting its forever loop.

I've mostly resolved all the issues now. It all had to do with timing and innappropriate use of global variables. Or at least that's my theory.

The reason why the character array `startup_log` was showing up in an incomplete form with all those NULL values at the end, is because the SD card task was writing `startup_log` while a bunch of other tasks were modifying it, which may have led to it being corrupted. 

Then there's the debug messages repeating. In that last example I showed, this was printed to the debug log:
```
12:53:02 03/12/2025    OLED page is now 1
12:53:02 03/12/2025    OLED page is now 1
```

However, what was printed in the terminal was:
```
12:53:02 03/12/2025    Button 26 pressed!
12:53:02 03/12/2025    OLED page is now 1
```

What's happening here is that two debug messages were sent in quick succession. By the time the SD Card Task was ready to print that first debug message into the text file, the value of `debug_buffer` had already been changed to the most recent message. Therefore, the same debug message got printed twice in the text file.

So here's how I fixed it. I did what I should have done from the very beginning but was somehow too lazy to: QUEUES!

It should have been a no-brainer. I've got a bunch of debug messages but they need to wait until the SD Card Task is ready before they can be saved to the text file. What's a perfect place for data to wait? QUEUES!

I initialize it in `debug_uart_init()`:
```c

void debug_uart_init(int pio_tx_pin){
    bool success = pio_claim_free_sm_and_add_program_for_gpio_range(&debug_uart_program, &debug_pio, &debug_sm, &debug_offset, pio_tx_pin, 1, true);
    hard_assert(success);

    debug_uart_program_init(debug_pio, debug_sm, debug_offset, pio_tx_pin, SERIAL_BAUD);
    
    debug_queue = xQueueCreate(256, sizeof(char)*256);

    debug_mutex = xSemaphoreCreateMutex();
}
```

Then in `debug_print()`, after `debug_buffer` has been filled with the formatted debug message:
```c
void debug_print(const char* fmt, ...){
    xSemaphoreTake(debug_mutex, portMAX_DELAY);

    memset(debug_buffer, 0, sizeof(debug_buffer));

    char timestamp[50];
    memset(timestamp, 0, sizeof(timestamp));

    if (rtc_initialized){
        //Copy timestamp string into variable timestamp
        strcpy(timestamp, rtc_get_timestamp_string());
        sprintf(debug_buffer, timestamp);
        newline = 0;
    }

    va_list list;
    va_start(list, fmt);

    vsprintf(debug_buffer + strlen(timestamp), fmt, list);
          
    va_end(list);
    
    debug_uart_program_puts(debug_pio, debug_sm, debug_buffer);


    xQueueSendToBackFromISR(
                    debug_queue, 
                    (void*)&debug_buffer, 
                    NULL);
    
    if (startup_done){
        xTaskNotify(sd_writeTaskHandle, 0, eNoAction);
    }

    xSemaphoreGive(debug_mutex);
}
```

Every debug message is always sent to the queue. If the SD Card Task has finished intialization, a notification is sent to it so it reads and empties the entire queue, as shown below:

```c
//First, write debug messages from startup into text file
//Open file
pxFile = ff_fopen(filename, "a");
if (!pxFile) {
    stop();
}

char buffer[256];
//Read and empty the entire queue into file
while(uxQueueMessagesWaiting(debug_queue) != 0){
    memset(buffer, 0, sizeof(buffer));  
    
    xQueueReceive(
        debug_queue,
        &buffer,
        0
    );

    if (ff_fprintf(pxFile, buffer) < 0) {
        stop();
    }
}

//Close file    
if (-1 == ff_fclose(pxFile)) {
    stop();
}

debug_print("SD Startup done!\r\n");
startup_done = 1;
for(;;){
    xTaskNotifyWait(0,0, NULL, portMAX_DELAY);
    //Open file
    pxFile = ff_fopen(filename, "a");
    if (!pxFile) {
        stop();
    }

    //Read and empty the entire queue into file
    while(uxQueueMessagesWaiting(debug_queue) != 0){
        memset(buffer, 0, sizeof(buffer));
        
        xQueueReceive(
            debug_queue,
            &buffer,
            0
        );

        if (ff_fprintf(pxFile, buffer) < 0) {
            stop();
        }
    }
    //Close file
    if (-1 == ff_fclose(pxFile)) {
        stop();
    }
}
```
