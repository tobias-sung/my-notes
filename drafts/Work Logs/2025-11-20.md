Here's a summary of the hardware registers for each I2C instance:
Key I2C Hardware Registers on the RP2040:

The I2C functionality is controlled by a set of registers, often defined with offsets from a base address in the SDK's header files (e.g., `hardware/regs/i2c.h`). The primary registers include: 

- **`IC_CON` (I2C Control Register):** Configures the operational mode of the I2C peripheral (e.g., master/slave mode, speed, restart conditions, I2C enable/disable). This register can only be written when the I2C block is disabled.
- **`IC_DATA_CMD` (I2C Rx/Tx Data Buffer and Command Register):** This is the main register the CPU reads from (RX FIFO) and writes to (TX FIFO) for data transfer. It also includes bits to control whether a `STOP` or `RESTART` condition is issued after the byte is sent/received.
- **`IC_SAR` (Slave Address Register):** Holds the slave address used when the I2C block is configured to operate in slave mode.
- **`IC_TAR` (Target Address Register):** Holds the target slave address when the I2C block is configured as a master.
- **`IC_RXFLR` and `IC_TXFLR` (RX/TX FIFO Level Registers):** Provide the current fill level of the receive and transmit FIFOs, respectively.
- **`IC_STATUS` (I2C Status Register):** Contains various status flags, such as bus activity, FIFO full/empty status, and transmit/receive errors.
- **`IC_RAW_INTR_STAT` / `IC_INTR_STAT` (Interrupt Status Registers):** Indicate pending interrupts (e.g., RX full, TX empty, transmit abort).
- **`IC_SS_SCL_HCNT` / `IC_SS_SCL_LCNT` (Standard Speed SCL High/Low Count Registers):** These registers determine the high and low periods of the SCL clock signal for standard speed I2C communication, used for setting the bus speed.

Learned that `tight_loop_contents()` is a No-Op function (i.e. does nothing) meant to run in a loop. It's defined in `platform.h` in the Pico SDK. 
****
At least now I know why the I2C address is showing up as 0x29. It's because the address is calculated as follows:
```c
Dev.platform.address = (uint8_t)((VL53L5CX_DEFAULT_I2C_ADDRESS >> 1) & 0xFF);
```
where `VL53L5CX_DEFAULT_I2C_ADDRESS` is equal to 0x52. Somehow, the bit shift and AND operation lead to its value being 0x29. Found this out by printing the address out in HEX form (i.e. `print("%X",Dev.platform.address)`).

I wanted to understand what exactly is happening during a VALID read/write operation to/from the sensor. So I decided to record the messages being transported in `vl53l5cx_is_alive()` using PulseView (with a sampling rate of 8MHZ). Note that the master is the Pico and the slave is the sensor.

I've been referring to this [handy manual](https://www.ti.com/lit/an/slva704/slva704.pdf?ts=1763596658044&ref_url=https%253A%252F%252Fwww.ti.com%252Fproduct%252FTCA6416A%253FkeyMatch%253DTCA6416A%2526tisearch%253Duniversal_search%2526usecase%253DGPN-ALT) from Texas Instruments on how SPI works.

![[Pasted image 20251120164808.png]]
1. Write value 0 to register address 7FFF. So we start with a START signal to indicate that a transfer is about to begin (a START is indicated by having SDA go from HIGH to LOW while SCL is HIGH). Then we send the slave address (the i2C address of the sensor) 0x29, then the register address 0x7FFF, then finally the data 0x00. The transmission is terminated by a STOP signal (indicated by having SDA go frm LOW to HIGH while SCL is HIGH).  
 
![[Pasted image 20251120165036.png]]
Then, read a byte from register address 0. 

Before the master Pico can read from the slave sensor, it needs to tell the slave which register it wishes to read from. 

Therefore, every read operation starts by writing the register address to the slave address (so in this case, we write `{00,00}` to 0x29 with the Read/Write bit set to 0, meaning Write). Once the slave acknowledges that it received the register address, master sends a REPEAT START signal to indicate the start of a new tranmission. 

It then sends slave address with the Read/Write bit set to 1 (meaning Read). Once this is acknowledged by the slave, the master releases the SDA bus to the slave, so the slave can send data. After every byte of data, if the master expects more it will send an ACK. Once it has received the expected number of bytes, it will send a NACK followed by a STOP condition. The slave then gives up the SDA bus.

Read a byte from register address 1. Same procedure as above.

Finally, write value 2 to register address 7FFF.

Writing this a day later. As always, I found my solution online (I didn't expect to find one because I thought this would be a niche issue). However, if it weren't for using the Logic Analyzer I wouldn't have known how to phrase the issue, that being the fact that the I2C read was incorrect while the Logic Analyzer output was correct. 

Here's the [link](https://forums.raspberrypi.com/viewtopic.php?t=340111). I have no idea how he came to this conclusion (hopefully I'll be able to read the entire post), but he simply made the following modification to the PIO program:
```
    ; Handle ACK pulse
    out pindirs, 1         [7] ; On reads, we provide the ACK.
    nop             side 1 [7] ; SCL rising edge
    wait 1 pin, 1          [7] ; Allow clock to be stretched
    jmp pin do_nack side 0 [2] ; Test SDA for ACK/NAK, fall through if ACK
    mov isr, null              ; Reset the input bit counter

public entry_point:
.wrap_target
```

He added a `mov isr, null` at the end of the so-caled "byte loop".
